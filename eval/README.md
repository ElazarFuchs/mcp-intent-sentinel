# Reproducing the MIS measurement

This directory holds the evaluation harness. It exists so anyone who reads
[the v0.1.5 report](results/v0.1.5/report.md) can reproduce every number in
it — including the ones that make MIS look bad.

## What's in here

```
eval/
├── README.md          ← you are here
├── registry.py        ← the 51-server list scanned in v0.1.5
├── run.py             ← the harness: download → scan → write report
└── results/
    └── v0.1.5/
        ├── report.md  ← human-readable summary (verdict %s, FP list, etc.)
        └── run.json   ← raw per-server record: verdict + tools + findings + timings
```

Past run snapshots live alongside `v0.1.5/` — `v0.1.3/`, `v0.1.4/` — so the
delta between MIS versions is auditable. If a number in any report changes
across a re-run, the run.json files are the ground truth.

## What gets measured

For each entry in `registry.py` the harness:

1. **Downloads** the package via the relevant scheme (`npm:` → `npm pack`,
   `pypi:` → `pip download --no-deps --no-binary=:all:`, `github:` → shallow
   clone). No install scripts execute — `npm pack` and `pip download` are
   fetch-only. See [LIMITATIONS.md L12](../LIMITATIONS.md) for what schemes
   are NOT supported.
2. **Scans** the extracted source via `mis.engine.scan()` — same code path as
   the user-facing `mis scan <source>` CLI.
3. **Records** verdict, confidence, tool count, tool names, behavior
   extraction count, finding count, top rule hits, and timing.

The harness never executes the server. Static analysis only — see
[LIMITATIONS.md L1](../LIMITATIONS.md) for what that means and doesn't mean.

## Run it

Requires Python ≥ 3.11, `pip install -e .` from the repo root, and:
- **`npm` on PATH** for npm: entries (Node 18+ recommended). Without it,
  every npm entry shows up in the "Failed downloads" section.
- **`pip` on PATH** (you have it if you're reading this).

```bash
# Full run on the 51-server registry. ~5–10 minutes depending on network.
python -m eval.run

# Output goes to eval/results/latest/. Set --out to keep prior runs around.
python -m eval.run --out eval/results/my-rerun

# Smoke test on the first N entries.
python -m eval.run --limit 5 --out eval/results/smoke

# Resume an interrupted run.
python -m eval.run --resume --out eval/results/my-rerun
```

## Reading the report

`results/<run>/report.md` is generated by `run.py` and breaks down into:

- **Verdict distribution** — counts and percentages over the *successfully
  scanned* set (failed downloads are tallied separately).
- **FP candidates** — every server MIS flagged `malicious` or `suspicious`.
  Each requires manual review: either a real find to disclose upstream, or
  an over-aggressive rule to tighten. The list is always printed.
- **Shallow list** — servers where MIS detected tools but extracted no
  behavior from any of them. These are MIS's blind spots, named.
- **Unknown list** — SDK registration patterns MIS doesn't recognize at all.
  This is the L13 coverage gap, by name.
- **Benign list** — split into two sub-lists:
  - **with extracted behavior**: MIS read at least one tool's body and the
    rules didn't fire. Real benign candidates, bounded by L4.
  - **zero behavior extracted**: ⚠️ ZERO ENTRIES expected. If any appear,
    that's a regression — the v0.1.5 `_has_io_capable_imports` fix leaked.
- **Failed downloads** — registry entries that couldn't be fetched (renamed,
  retired, archive errors). Reported by name with the error string so the
  denominator can't be silently skewed.

## Reproducing the v0.1.5 headline numbers

```text
33 / 51 servers scanned successfully (18 failed download)

  benign     12.1% (4)   ← all with extracted behavior; 0 leak
  shallow    24.2% (8)
  unknown    54.5% (18)
  suspicious  6.1% (2)
  malicious   3.0% (1)
```

If your run differs from these numbers:

- **Package versions moved.** npm and PyPI shift; a server may have been
  re-published with a different shape. `registry.py` pins names, not
  versions — by design, so we measure today's ecosystem, not a frozen one.
- **Network conditions.** `npm pack` and `pip download` time out at 120s
  per package; flaky networks shift entries into "Failed downloads".
- **Newer MIS detectors.** If a future MIS version learns a new SDK
  pattern, the unknown count drops and the benign/shallow/etc. counts
  shift accordingly. The whole point of keeping `results/<version>/`
  separate is to make this delta auditable.

## Adding a server to the registry

Edit `registry.py`. Each entry is a `ServerEntry(name, source, ecosystem,
expected_class, notes)`. The `source` string is whatever you'd pass to
`mis scan` directly. Re-run the harness — your new entry shows up in the
distribution and (if applicable) in the appropriate sub-list.

**Don't curate.** The point of the registry is to be a representative
sample of what people actually install. Removing a server because it makes
MIS look bad would defeat the measurement.

## Comparing against other scanners

`mcp-scan` (now `snyk-agent-scan`) is the only OSS MCP scanner with broad
adoption. It scans MCP **config files and runtime tool descriptions**, not
server source — so a head-to-head against MIS is comparing apples to
oranges. See [LIMITATIONS.md L10](../LIMITATIONS.md) for the longer note.
The `--baseline` flag in `run.py` invokes mcp-scan in a stub mode; a real
side-by-side that compares the two on the same artifact requires more
plumbing and is tracked in [ROADMAP.md](../ROADMAP.md) for v0.2.

## If MIS is blind somewhere

That's the most useful thing you can tell me. Open an issue with the
package name (or a minimal source snippet) and the verdict you got. The
eval harness exists so we can argue from numbers, not from intuition.
